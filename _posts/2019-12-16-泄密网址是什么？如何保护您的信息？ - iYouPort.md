  <p class="graf graf--p">
   泄密网址是什么？
  </p>
  <p class="graf graf--p">
   URL是最常见的跟踪信息。
  </p>
  <p class="graf graf--p">
   根据页面内容构建URL这种看起来无辜的选择可以使曝光用户的浏览历史记录、地址、健康信息、甚至更敏感的详细个人信息变得更加容易。它们本身就包含敏感信息或可能引导至包含敏感信息的页面。
  </p>
  <p class="graf graf--p">
   我们称这种网址为泄密网址。
  </p>
  <p class="graf graf--p">
   以下是这类URL的一些例子。
  </p>
  <p class="graf graf--p">
   <strong class="markup--strong markup--p-strong">
    示例＃1：
   </strong>
  </p>
  <p class="graf graf--p">
   Website: donate.mozilla.org (Fixed)
  </p>
  <p class="graf graf--p">
   在 donate.mozilla.org 上完成付款流程后，您将被重定向到“感谢”页面。
  </p>
  <p class="graf graf--p">
   如果您仔细查看下面的屏幕截图中显示的网址，你会发现，它会包含一系列私人信息：如 电子邮件、国家/地区、金额、付款方式。
  </p>

    在这里找到
   </a>
   。
  </p>
  <p class="graf graf--p">
   <strong class="markup--strong markup--p-strong">
    示例＃2：
   </strong>
  </p>
  <p class="graf graf--p">
   Website: trainline.eu, JustFly.com (Last checked: Aug’18)
  </p>
  <p class="graf graf--p">
   完成购买火车票/机票的在线操作后，您会收到一封电子邮件，其中包含管理您预订的链接。
  </p>
  <p class="graf graf--p">
   大多数情况下，当您点击链接时，会显示预订详细信息 —— 无需输入任何其他详细信息了，如预订代码，用户名/密码。
  </p>
  <p class="graf graf--p">
   这意味着该URL本身包含令牌，该令牌对用户是唯一的，以此提供对用户预订的访问。
  </p>
  <p class="graf graf--p">
   碰巧这些URL也与第三方共享，⚠️为这些第三方提供高度敏感的数据
   <a class="markup--anchor markup--p-anchor" data-href="https://cliqz.com/en/magazine/lufthansa-data-leak-what-a-single-url-can-reveal-about-you" href="https://cliqz.com/en/magazine/lufthansa-data-leak-what-a-single-url-can-reveal-about-you" rel="noopener noreferrer" target="_blank">
    以访问您的预订
   </a>
   。
  </p>

   要明确的是，不仅仅是上述这些网站遭受此类泄密。
  </p>
  <p class="graf graf--p">
   这个问题几乎无处不在 —— 这是一种默认情况，而不是罕见的情况。
  </p>
  <p class="graf graf--p">
   <strong class="markup--strong markup--p-strong">
    泄密网址的风险
   </strong>
   ：
  </p>
  <ul class="postList">
   <li class="graf graf--li">
    网站不小心将敏感信息泄露给了过多的第三方；
   </li>
   <li class="graf graf--li">
    大多数情况下未经用户同意；
   </li>
   <li class="graf graf--li">
    更危险的是：大多数网站仍没有意识到这些泄漏。
   </li>
  </ul>
  <p class="graf graf--p">
   这些问题难以修复吗？
  </p>
  <p class="graf graf--p">
   我理解使用第三方服务来优化和增强数字产品以及增强用户与产品交互的必要性。
  </p>
  <p class="graf graf--p">
   但所有者应始终控制其网站以及该网站与第三方服务共享的内容。需要执行此控制来限制用户信息的泄漏。
  </p>
  <p class="graf graf--p">
   这并不是一个庞大的任务，只是致力于保护隐私权的基本措施。
  </p>
  <p class="graf graf--p">
   例如：
  </p>
  <ol class="postList">
   <li class="graf graf--li">
    私人页面应该有noindex元标记。
   </li>
   <li class="graf graf--li">
    限制私人页面上存在第三方服务。
   </li>
   <li class="graf graf--li">
    在具有敏感数据的页面上使用 Referrer-Policy。
   </li>
   <li class="graf graf--li">
    实施CSP和SRI。目前多数网站都没有启用SRI。
   </li>
  </ol>
  <p class="graf graf--p">
   <strong class="markup--strong markup--p-strong">
    介绍 Local Sheriff
   </strong>
  </p>
  <p class="graf graf--p">
   众所周知此类信息泄露对用户和组织来说都是非常危险的，但为什么它仍然是一个广泛存在的问题呢？
  </p>
  <p class="graf graf--p">
   存在这些问题的一个重要原因是缺乏安全意识。
  </p>
  <p class="graf graf--p">
   作为网站，一个很好的起点是查看哪些信息会被泄露、或检测到泄密URL的存在。
  </p>
  <p class="graf graf--p">
   但是，为了查明您维护或访问的网站是否发生了同样的情况，您需要掌握一些基本工具。
  </p>
  <p class="graf graf--p">
   我们希望使用以下准则来构建这类工具：
  </p>
  <ul class="postList">
   <li class="graf graf--li">
    易于安装。
   </li>
   <li class="graf graf--li">
    数据在用户计算机上本地储存。
   </li>
   <li class="graf graf--li">
    帮助识别公司在互联网上跟踪用户的行为。
   </li>
   <li class="graf graf--li">
    可搜索泄露给第三方的信息的界面。
   </li>
  </ul>
  <p class="graf graf--p">
   鉴于上述指导原则，浏览器扩展似乎是一个合理的选择。在后台安装 Local-Sheriff 之后：
  </p>
  <ol class="postList">
   <li class="graf graf--li">
    使用 WebRequest API，它可以监视第一方和第三方之间的交互。
   </li>
   <li class="graf graf--li">
    对URL进行第一方和第三方分类。
   </li>
   <li class="graf graf--li">
    <a class="markup--anchor markup--li-anchor" data-href="https://whotracks.me/" href="https://whotracks.me/" rel="noopener noreferrer" target="_blank">
     WhoTracksMe
    </a>
    数据库副本。映射哪个域属于哪个公司。
   </li>
   <li class="graf graf--li">
    提供一个界面，您可以搜索您认为属于您的值，并查看哪些网站将其泄露给哪些第三方。例如：姓名、电子邮件、地址、出生日期、cookie等。
   </li>
  </ol>
  <p class="graf graf--p">
   <strong class="markup--strong markup--p-strong">
    重温示例＃1
   </strong>
  </p>
  <p class="graf graf--p">
